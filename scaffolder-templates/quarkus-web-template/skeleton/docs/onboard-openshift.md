# OpenShift에 개발자를 온보딩하기 위한 가이드

컨테이너와 OpenShift에 개발자 및 플랫폼 엔지니어를 온보딩하기 위한 가이드입니다.

## 소개

플랫폼 팀은 OpenShift 클러스터를 구축했습니다. 클러스터는 워크로드를 할당할 준비가 되어 있습니다. 개발자들은 애플리케이션을 마이그레이션하고자 하지만, 이를 위한 현재 프로세스가 필요합니다. 이 문서는 온보딩 프로세스를 설계할 때 사용할 수 있는 템플릿을 제공합니다. 그대로 사용하셔도 됩니다.

### 개발자 팀 가이드: 현대화 여정을 시작하기 위한 개발자의 출발점

개발자 팀이 컨테이너 여정을 시작하기 위해 준비하는 프로세스입니다.

1. 개발자 기초 학습: 학습 과정 따르기
2. 애플리케이션 평가: 컨테이너를 위한 Twelve-Factor App 적용하기

[컨테이너를 위한 Twelve-Factor App 핵심 체크리스트]()

애플리케이션 소유자와 팀은 애플리케이션의 성숙도 수준, 위험 요소, 잠재적 변경사항 및 구현 가능성을 파악하기 위해 고수준의 애플리케이션 평가를 수행합니다.

- [ ] I. 코드베이스: 하나의 코드베이스를 버전 관리 시스템으로 추적하고 여러 배포 사용
- [ ] II. 종속성: 종속성을 명시적으로 선언하고 격리
- [ ] III. 구성: 환경에 구성 저장
- [ ] V. 빌드, 릴리스, 실행: 빌드 단계와 실행 단계를 엄격히 분리
- [ ] XI. 로그: 로그를 이벤트 스트림으로 취급
- [ ] 보안 및 규정 준수: 적용해야 할 보안 및 규정 준수를 이해

3. OpenShift 접근 권한 요청

조직에서 필요하다면, 애플리케이션과 개발자 팀 정보를 수집하기 위한 자동화된 반복 가능한 프로세스를 만드세요.

### 플랫폼 팀 체크리스트

이 프로세스는 애플리케이션 및 팀의 요구사항을 고려하여 개발자 팀과 애플리케이션에 OpenShift 접근 권한을 부여하는 플랫폼 팀의 절차를 설명합니다.

#### 1단계 프로젝트 설정 (플랫폼)

이 프로세스는 개발자 팀과 애플리케이션의 요구사항에 따라 필요한 모든 리소스를 생성합니다.

##### 플랫폼 팀 체크리스트

개발자는 새로운 네임스페이스, 해당 네임스페이스에 대한 접근 권한 및 애플리케이션의 동작에 필요한 기타 요구사항이 필요합니다.

아래는 팀과 애플리케이션을 OpenShift에 성공적으로 온보딩하기 위한 플랫폼 가이드를 만드는 데 추천되는 목록입니다.

- [ ] 1- 팀별 프로젝트 프로비저닝
  - [ ] 하위 환경을 위한 네임스페이스 생성
    - [ ] 각 하위 환경과 레지스트리에 사용자 접근 권한 부여
    - [ ] 프로젝트 설정
    - [ ] 리소스 관리
- [ ] 2- OpenShift에서 애플리케이션 구성
  - [ ] 애플리케이션 접근 권한 (팀이 인바운드/아웃바운드 종속성을 지정한 경우)
  - [ ] 회사 가이드라인에 따른 네임스페이스별 네트워크 정책
  - [ ] 애플리케이션 라우트 정의
  - [ ] 구성 요소/종속성별 방화벽 규칙 설정
- [ ] 3- CI/CD
  - [ ] CI/CD 접근 권한 (예: Jenkins/OpenShift Pipelines)
  - [ ] 회사 레지스트리에 대한 OpenShift 접근 권한 예: [Quay 컨테이너 레지스트리](https://quay.io/)
- [ ] 4- OpenShift에서 필요한 통합
  - [ ] 회사 애플리케이션 로깅 툴 접근 권한
    - [ ] 애플리케이션용 신규 인덱스 생성
  - [ ] 회사 관측 및 모니터링 툴링 접근 권한
- [ ] 5- 보안
  - [ ] 애플리케이션에 필요한 인증/권한/정책 툴 접근 권한
    - [ ] 신규 정책 생성
  - [ ] 인증서 생성 정의 및 가이드라인 공유
  - [ ] 회사 Vault 접근 권한
  - [ ] 인증서 저장소 접근 권한
    - [ ] 신규 인증서 생성
- [ ] 6- 개발자에게 환영 이메일 발송

#### 2단계 OpenShift 환영

사용자에게 확인 메일을 전송합니다. 개발자는 컨테이너를 위한 개발자 가이드를 따릅니다.
